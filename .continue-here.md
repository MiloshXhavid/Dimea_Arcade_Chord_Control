## Suspected root cause (Single-Channel mode + noteCount_ dedup)

With Single-Channel ON, all voices share one sentChannel_. If two voices play the SAME pitch simultaneously:
- noteCount_[ch][pitch] increments to 2
- First pad release: count 2→1, NO note-off sent (correct — other voice still held)
- Second pad release: count 1→0, note-off sent

If there's ANY mismatch between note-on pitch (at press) and note-off pitch (at release) — e.g. joystick moved between press and release for one voice — count can get stuck at 1 permanently.

The activePitch_[v] snapshot should prevent this... but there may be an edge case where a re-trigger fires for voice V while pad is held, overwriting activePitch_[v], so the subsequent release sends note-off for the NEW pitch (count 1→0) but the OLD pitch count stays at 1 forever.

## Suspected bugs to fix

### Fix A — Looper stop resetNoteCount() (lines 824, 856) [NOT current issue but fix anyway]
Remove `resetNoteCount()` from both looper stop/reset paths. Targeted per-voice decrements above already handle this.

### Fix C — Re-trigger while pad held overwrites activePitch_[v]
In TriggerSystem::fireNoteOn, when a note-on fires for voice V whose pad is already held (re-trigger on joystick move), activePitch_[v] is overwritten. On release, note-off targets the new pitch only — old pitch count leaks.

Possible fix: only fire note-off for old pitch first (if activePitch_[v] != newPitch && gateOpen_[v]) before overwriting activePitch_[v].
