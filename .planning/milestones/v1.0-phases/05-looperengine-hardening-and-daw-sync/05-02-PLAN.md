LOOPER GATE PLAYBACK — LIVE INPUT PASSTHROUGH MODEL:
The locked decision (CONTEXT.md Section B) requires: "Live gate triggers always pass through to MIDI output even while the looper is playing back. The looper does not block live input."

To satisfy this without conflict, looper gate events BYPASS TriggerSystem entirely and emit MIDI directly. This means: when loopOut.gateOn[v] is true, look up the current heldPitch_[v] and the voice's MIDI channel, then call midi.addEvent() directly. When loopOut.gateOff[v] is true, emit noteOff directly. Do NOT call trigger_.setPadState() for looper gate events.

Replace the existing "Looper gate overrides" block (which incorrectly calls trigger_.setPadState):
  // REMOVE THIS BLOCK:
  // for (int v = 0; v < 4; ++v) {
  //   if (loopOut.gateOn[v])  trigger_.setPadState(v, true);
  //   if (loopOut.gateOff[v]) trigger_.setPadState(v, false);
  // }

  // REPLACE WITH (after voiceChs[] is populated, before trigger_.processBlock()):
  // Looper gate playback: emit MIDI directly, bypassing TriggerSystem.
  // This satisfies the locked decision: live pad input passes through independently
  // because TriggerSystem is not informed of looper events — it only sees live pad state.
  for (int v = 0; v &lt; 4; ++v) {
    const int ch0 = voiceChs[v] - 1;  // 0-based for MIDI message, +1 inside addEvent
    if (loopOut.gateOn[v]) {
      midi.addEvent(juce::MidiMessage::noteOn(ch0 + 1, heldPitch_[v], (uint8_t)100), 0);
    }
    if (loopOut.gateOff[v]) {
      // Use trigger_.getActivePitch(v) if available for the note-off pitch,
      // otherwise fall back to heldPitch_[v] (must match the note-on pitch)
      midi.addEvent(juce::MidiMessage::noteOff(ch0 + 1, heldPitch_[v], (uint8_t)0), 0);
    }
  }

NOTE: voiceChs[] must be populated BEFORE the looper gate emission block. In the current processBlock structure, voiceChs[] is populated before the looper gate overrides block — maintain this ordering.
