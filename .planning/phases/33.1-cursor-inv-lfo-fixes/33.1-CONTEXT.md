# Phase 33.1: Cursor, INV, LFO, Piano, Battery Bug Fixes — Context

**Gathered:** 2026-03-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix 7 bugs discovered after Phase 32. No new features, no APVTS parameter additions.

1. Cursor starts at wrong position on plugin open (center-X, bottom-Y instead of center)
2. Spring-back animation too jumpy/oscillating when cursor returns to center
3. LFO rate modulation via left stick is asymmetric (right direction much smaller range than left)
4. INV mode: red X/Y signs in Quantizer box (drawn in paint()) do not swap
5. INV mode: modulation box incomplete swap — red "Left X"/"Left Y" paint labels don't swap, and combo box + knob attachments don't swap (wrong APVTS params stay wired to wrong rows)
6. Scale piano keyboard: clicking black keys sometimes registers the white key underneath
7. Gamepad battery level icon shows wrong charge level

</domain>

<decisions>
## Implementation Decisions

### Bug 1 — Cursor initial position
- **Root cause:** `displayCx_/Cy_` member-initializes to `0.0f`. On plugin open the joystick Y maps to bottom-center visually (coordinate system or atomic default issue). The cursor visually renders at wrong position until first timerCallback fires.
- **Fix:** In `JoystickPad::resized()` (or `PluginEditor::resized()` wherever displayCx_/Cy_ live), snap `displayCx_/Cy_` to the pad's center pixel coordinates immediately — do not wait for first timerCallback tick. Also initialize `springVelX_/Cy_` to 0.
- **Result:** Cursor always appears at center on plugin open, then smoothly follows actual joystick position.

### Bug 2 — Spring-back too jumpy
- **Root cause:** `kDamping = 0.72f` is underdamped — causes visible overshoot and oscillation on release.
- **Fix:** Increase `kDamping` to `0.90f` (near-critical damping). Cursor settles smoothly with no bounce.
- `kSpring = 0.18f` stays unchanged — settle speed remains ~90ms.
- **Do not** change to lerp/exponential smoothing — keep spring physics, just damp it properly.

### Bug 3 — LFO rate modulation asymmetric
- **Root cause:** `base + stick * 9.995f` applies a linear offset to a log-scaled parameter (0.01–20 Hz, skew 0.35). Going left feels huge, going right feels tiny.
- **Fix:** Apply offset in normalized log space — symmetric in perceptual terms:
  ```cpp
  const float logMin = std::log(0.01f);
  const float logMax = std::log(20.0f);
  const float norm   = (std::log(std::max(0.01f, base)) - logMin) / (logMax - logMin);
  const float actual = std::exp(logMin + juce::jlimit(0.f, 1.f, norm + stick * 0.5f) * (logMax - logMin));
  ```
- Applies to **both** `filterXMode==4` (LFO-X Freq) and `filterYMode==4` (LFO-Y Freq) code paths.
- The LFO rate slider itself is **unchanged** — only the stick-offset math changes.
- Stick range: ±0.5 of normalized space (same feel as Phase ±180°/360° = ±0.5 of full range).

### Bug 4 — INV mode: Quantizer X/Y paint signs don't swap
- **Root cause:** The red "X" and "Y" axis divider marks in the Quantizer box are drawn statically in `paint()` — not Label components.
- **Fix:** Read `stickInvert` APVTS param in `paint()` (same `> 0.5f` check used in timerCallback). Conditionally swap which position gets "X" vs "Y" drawn.
- Use `*proc_.apvts.getRawParameterValue("stickInvert") > 0.5f` — safe to call from message thread in paint().

### Bug 5 — INV mode: Modulation box incomplete swap
Three things need fixing:

**5a — Red "Left X"/"Left Y" paint labels don't swap:**
- Same pattern as Bug 4: these labels are drawn in `paint()`, not Label components.
- Fix: read `stickInvert` in `paint()` and swap the drawn strings ("Left X"↔"Left Y").

**5b — Combo box attachments don't swap:**
- Currently: `filterXModeAttachment` always wires the X-row dropdown to `filterXMode` APVTS param, regardless of INV state.
- In INV mode, physical stick X controls logical Y — so the X-row dropdown should control `filterYMode` and vice versa.
- **Fix:** When `stickInvert` toggles, swap the ComboBox APVTS attachments:
  - Detach `filterXModeAttachment_` and `filterYModeAttachment_`
  - Reattach X-row combo to `filterYMode` param (and Y-row to `filterXMode`) when INV ON
  - Reattach normally when INV OFF
- Trigger this swap in `timerCallback()` when `stickInvert` state changes (track previous state with a `bool prevInvState_` member).

**5c — Atten knob attachments don't swap:**
- Same pattern: `filterXAttenAttachment_` / `filterYAttenAttachment_` need to swap when INV toggles.
- Fix: same detach/reattach approach as 5b, triggered together in timerCallback on INV state change.

**Swap trigger:** A single `bool prevInvState_` member in PluginEditor tracks the last known INV state. On each timerCallback, compare current `stickInvert` value to `prevInvState_`. On change: detach all 4 attachments, reattach swapped or normal, update `prevInvState_`.

### Bug 6 — Piano keyboard: black key clicks register as white key
- **Root cause:** `ScaleKeyboard::mouseDown()` iterates notes 0–11 in chromatic order. Black key C# (note 1) overlaps white key C (note 0) in X. A click at X=0.7*wKey falls inside C's full-height rect and matches first — loop returns early, never reaches C#.
- **Fix:** Two-pass hit-test: first pass checks only black keys (notes 1,3,6,8,10), returns on match. Second pass checks white keys (0,2,4,5,7,9,11). Black keys always win over overlapping white keys.
- No changes to `noteRect()` geometry — only the loop order in `mouseDown()` changes.

### Bug 7 — Battery level shows wrong charge
- **Root cause:** SDL2's `SDL_JoystickCurrentPowerLevel()` on Windows returns inaccurate data for PS4/PS5 controllers — commonly returns `FULL` (3) regardless of actual charge, or `WIRED` (4) when wireless.
- **Fix approach:** Researcher to verify if SDL2 2.30.9 has a more reliable API for PS4/PS5 battery on Windows.
- **Fallback if no better API:** Show battery icon as "?" (unknown) by default; only display a definite level when SDL reports `EMPTY` or `LOW` (warning states are reliable even if MEDIUM/FULL aren't). Prevents false "full" readings from misleading the user.
- Do NOT remove the battery icon — low/empty warnings still provide value.
