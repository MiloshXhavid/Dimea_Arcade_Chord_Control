# Phase 43.1 — Mini Mode: Context

## Goal
A 3-state window mode toggle: Full UI ↔ Mini (pad-only small) ↔ Maxi (pad-only fills screen). In both Mini and Maxi, only the joystick pad renders — starfield, warp, cursor. All controls hidden. Intended use: park the widget in a DAW corner (Mini) or use the joystick as a fullscreen performance surface (Maxi).

---

## Decisions

### 1. Window Mode States

Three states, cycled via header bar buttons:

| State | Window size | Content |
|-------|-------------|---------|
| **Full** | Normal (1120 × 840 × scale) | Everything |
| **Mini** | Pad square at current scale (~308 × scale) | Pad only |
| **Maxi** | Pad square filling available display | Pad only |

**Enter Mini:** ◱ button far-right of header bar (~20×20px, pixel-art style, dim until hover). No header visible in Mini — overlay button in top-right corner of pad (semi-transparent, always visible) cycles back to Full.

**Enter Maxi:** Second button in header bar next to ◱ — an expand/fullscreen icon (⛶ or □→⬛). In Maxi, same overlay button approach as Mini (top-right corner) to return to Full.

**Cycle:** Full → Mini and Full → Maxi are direct via header buttons. Mini ↔ Maxi is not a direct transition — user always returns to Full first (via overlay button), then picks the other mode.

### 2. Size Details

**Mini:** Window = pad square at active scale factor
- At 0.75x → ~231×231 px
- At 1.0x → ~308×308 px
- At 2.0x → ~616×616 px

**Maxi:** Window = largest square that fits the usable display area
- Detected via `juce::Desktop::getInstance().getDisplays().getDisplayContaining(centre).userArea`
- `maxiSize = min(userArea.getWidth(), userArea.getHeight())` — square, display-filling
- Host may constrain this further (Ableton embedded window vs. Reaper floating) — implementation uses `setSize` and lets the host clip if needed
- Scale factor from Phase 43 is **ignored** in Maxi — window is always display-sized

### 3. Content in Mini / Maxi
- **Shown:** JoystickPad only — starfield, warp stars, spring-damper cursor, crosshair lines (if enabled), burst particles
- **Hidden:** Header bar, LFO panels, left column (TouchPlates, Scale, Quantizer, Trigger, Random, Looper, Arp), right column controls, footer, chord name label, logo
- In Maxi: JoystickPad `setBounds(0, 0, windowSize, windowSize)` — pad fills the entire window

### 4. Persistence
- Window mode state is **not** saved across sessions
- Plugin always opens in Full view regardless of how it was closed
- Scale factor (Phase 43) is saved separately and only applies to Full mode

### 5. Gamepad Option Mode Integration

The gamepad Option button cycle is extended from 3 → 5 states:

| optMode | Label | D-pad | View |
|---------|-------|-------|------|
| 0 | OPTION | BPM ↑↓ + looper rec ←→ | Full |
| 1 | ARP | octave steps | Full |
| 2 | INTRVL | transpose/intervals | Full |
| 3 | MINI | BPM ↑↓ + looper rec ←→ | Mini |
| 4 | MAXI | BPM ↑↓ + looper rec ←→ | Maxi |

**Cycle:** `(optionMode_ + 1) % 5` — change from `% 3` in GamepadInput.cpp.

**View switching:** The PluginEditor `timerCallback()` that reads `optMode` calls `applyWindowMode()` when the mode transitions into or out of 3/4. Mode 0/1/2 → Full; mode 3 → Mini; mode 4 → Maxi.

**D-pad in modes 3/4:** Reuses mode-0 looper behavior — change the D-pad dispatch condition in GamepadInput.cpp from `optionMode_ != 0` to `optionMode_ == 1 || optionMode_ == 2`. Modes 3/4 naturally fall through to the `else` branch.

**Overlay button sync:** Clicking the overlay button (↩, top-right corner in Mini/Maxi) returns to Full AND resets `optionMode_` to 0, keeping the cycle in sync.

**Label colours:** Mode 3 "MINI" = teal, Mode 4 "MAXI" = amber. (Labels are only visible during the brief moment before the header bar hides in Mini/Maxi — still set for completeness.)

## Code Context

### Header bar (paint, line ~4462)
- `auto header = getLocalBounds().removeFromTop(28)` — 28px, gradient fill, centered title text
- Button occupies far-right ~20×20px of this strip — no overlap with title (title is centred, leaves ~30px margin on each side at 1120px wide)

### JoystickPad bounds (resized, line ~3969)
- `joystickPad_.setBounds(padX, padTopY, padSize, padSize)` where `padSize ≈ 308` at 1x
- In mini mode: `setSize(padSize, padSize)` on the editor, `joystickPad_.setBounds(0, 0, padSize, padSize)`

### Implementation approach
- Add `enum class WindowMode { Full, Mini, Maxi }` + `WindowMode windowMode_` member to PluginEditor
- `applyWindowMode()` function: sets all non-pad components visible/hidden, calls `setSize` for the target dimensions
- Overlay button: drawn in `paintOverChildren` when `windowMode_ != Full` — semi-transparent filled rect + ↩ icon, top-right corner (24×24px, 4px inset)
- Mouse hit for overlay button: checked in `mouseDown` on PluginEditor (before forwarding to children)
- Header buttons: two small buttons at far-right of header strip — ◱ (Mini) and ⛶ (Maxi)
- Maxi size: `juce::Desktop::getInstance().getDisplays().getDisplayContaining(getScreenBounds().getCentre()).userArea` → take min dimension for square
- Phase 43 scale factor: `padSize = juce::roundToInt(308 * scaleFactor_)` for Mini; ignored for Maxi

### Components to hide in mini mode (all children except joystickPad_)
All TouchPlates, ScaleKeyboard, all knobs/sliders/combos/labels/buttons in left and right columns, LFO panels, chord name label, looper perimeter, footer text. The divider lines and logo are drawn in `paint()` — guard them with `if (!miniMode_)`.
