**Step 1 — processBlock zero-sample audit.**

Read `PluginProcessor::processBlock`. Verify:
- `audio.getNumSamples()` result is stored in `blockSize` but never used to index into `audio`
  (i.e. no `audio.getReadPointer(0)`, no `buffer.getSample(ch, i)` with i < blockSize guard,
  no division by `blockSize`)
- If any such access exists, wrap it with `if (blockSize == 0) return;` at the top of
  processBlock, immediately after `audio.clear()` and `midi.clear()`.

The current code stores `blockSize` and passes it to `trigger_.processBlock(tp)` via the
`TriggerParams` struct. Verify `TriggerSystem::processBlock` and `LooperEngine` code paths
that consume `blockSize` handle zero correctly (e.g., no division by blockSize, no
`for (int i = 0; i < blockSize; ++i)` that would still be safe but wasteful).

**Step 2 — Parameter listener balance audit.**

Read `PluginEditor.cpp`. Confirm:
- `ScaleKeyboard::ScaleKeyboard(...)` calls `apvts_.addParameterListener(...)` for
  "scalePreset", "useCustomScale", "globalTranspose", and "scaleNote0".."scaleNote11" (15 total)
- `ScaleKeyboard::~ScaleKeyboard()` calls matching `apvts_.removeParameterListener(...)` for
  all 15 IDs
- No other `addParameterListener` or `addListener` call exists in PluginEditor.cpp or
  PluginEditor.h that lacks a matching remove

If any unbalanced listener is found, add the missing `removeParameterListener` or
`removeListener` call in the appropriate destructor.

Open a standard cmd.exe window (not PowerShell — pluginval has documented compatibility


CRITICAL: No trailing backslash on the .vst3 path. Trailing slash causes "No types found"
failure (known pluginval bug, fixed in v1.0.4+ but defensive practice to omit it).

**Step 3 — Interpret results.**

If exit code 0 and all tests pass: DONE — proceed to Task 2 verification.

If exit code non-zero — fix issues immediately:

Common failure modes and fixes:
1. "No types found" → trailing slash on path, or installed path wrong. Re-check
   `C:\Program Files\Common Files\VST3\ChordJoystick.vst3` exists (exact spelling,
   no trailing slash in command).

2. "processBlock crash" → zero-sample buffer. Add guard at top of processBlock:
   ```cpp
   if (audio.getNumSamples() == 0)
       return;
   ```
   Rebuild Release, re-copy to VST3 path, re-run pluginval.

3. Parameter listener use-after-free crash → identify which listener is unbalanced
   (pluginval output will name the parameter or show a crash inside JUCE notification code).
   Add missing removeParameterListener in destructor. Rebuild + re-copy + re-run.

4. State save/restore failure (getStateInformation / setStateInformation) → usually
   XmlElement null dereference. Check PluginProcessor::getStateInformation and
   setStateInformation for null checks on ValueTree children.

5. If strictness-level 5 fails ONLY with VST3-SDK-validator audio-bus warnings (not
   reproducible at level 4) AND the error output references audio buses rather than a
   JUCE test name → fall back per CONTEXT.md policy:
   ```cmd
   pluginval.exe --validate-in-process --strictness-level 4 "C:\Program Files\Common Files\VST3\ChordJoystick.vst3"
   ```
   Document the fallback decision in the SUMMARY.

After any code fix: rebuild Release → re-copy bundle to `C:\Program Files\Common Files\VST3\`
→ re-run pluginval until exit code 0.
